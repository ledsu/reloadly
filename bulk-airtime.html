Bulk Airtime Scheduling

API & User Portal

A quick, easy, scalable API for distributing bulk airtime across 800 Telcos worldwide. Great use when telcos have scheduled promotions, sim GSM gateways, bulk orders for customers or taking top up orders from customers via csv file.

Schedule to process airtime top ups in bulk, with an easy API or directly from our web portal.

Processing bulk airtime is a feature that allows users to upload a csv file in 3 easy steps.

Upload the csv file with the numbers and amounts you want to top up.

Select a date and time to start the bulk processing and easily setup up the schedule and Reloadly will processing the file.

Fix or remove any errors returned from Reloadly, and finally submit and schedule your bulk airtime processing file.

After submitting your file, all successful transactions will be available to on your Transaction Report in real-time.

Open Source Bulk Airtime Application

Available for free on Github as a free open source application, pre-configured with Reloadlyâ€™s airtime API.
Programming Language: PHP Laravel (For backend) and Vue.JS (Frontend)
System Requirements: Docker + Make
Authentication: You will need to create a Reloadly Account, then update your API keys on the application.

As this is a scheduling app, it works in the backend so an active system is required. Users may choose to run this on their personal systems, however it is not recommended to do so. We recommend to get any cheap server/droplet and let it run there.

The system is fully based on Docker. The amazing world of Docker makes it possible to create a clean fresh environment on the cloud that has everything needed for the application to run. This gives developers freedom to deploy anywhere that Docker is compatible with. This application runs independently from Reloadlyâ€™s platform. It will simply call Reloadlyâ€™s API to process the airtime top ups with your existing API keys.

Port: As we do need a port of the system to have the application working. By default the system uses port 80 which can conflict in some situations. However the documentation on Github does explain how easy it is to simply edit a single text file and change port as needed.

Database Security: Users might have questions on how to keep the database secure that is why the database is NOT accessible to outside world. This is done by creating a virtual network on Docker, and then have the database and the PHP system deployed within that network. The system will only allow database connections to systems that are within that virtual network.

Default credentials: The default credentials are available in the documentation on the Github. However users are not required to change DB creds as the DB access is secured already. It is essential to change the default user's creds to run the app on a public server in order to avoid other people from accessing your system. This can be done simply within the system -> Settings. No need to mess with any files to do so.
